# Antimatter System Architecture

## 1. System Overview

The Antimatter ecosystem is a distributed, multi-component architecture designed to bridge a live Google AntiGravity IDE session running on a developer's host machine to a mobile companion application. By establishing a high-performance, bi-directional, and cryptographically secure WebSocket connection, Antimatter enables real-time observation, remote control, and synchronized state management of autonomous AI agents.

### 1.1 High-Level Topology

The system comprises three primary architectural pillars:
1. **The VS Code Extension (Host Server):** Acts as the authoritative state manager and proxy. It hooks deeply into the VS Code extension API, intercepting user commands, tailing local file logs generated by the AntiGravity agent, and managing the lifecycle of the secure WebSocket server.
2. **The Zero Trust Network Layer (Transport):** Rather than relying on insecure local network traversal or brittle third-party tunneling services (like LocalTunnel), Antimatter enforces strict zero-trust routing. It leverages Cloudflare's Edge network, utilizing either dedicated managed Cloudflare Tunnels (for enterprise deployments) or dynamically orchestrated Cloudflare Quick Tunnels (for zero-configuration developer onboarding).
3. **The Android Companion App (Client):** A native Android application built with modern declarative UI paradigms (Jetpack Compose). It acts as a stateless, reactive viewport and command center. It decodes the incoming JSON payload streams into interactive UI elements, rendering diffs, chat histories, and agent thoughts in real-time.

### 1.2 The Problem Domain

Developing autonomous coding agents introduces a severe observability challenge. Because agents can execute code, modify files, and spawn background tasks autonomously, developers must constantly monitor their progress. However, developers are not always stationed at their workstations. Antimatter solves the "remote observability problem" by unchaining the developer from their desk, allowing them to monitor complex, long-running agentic tasks from a mobile device without sacrificing the security of their host environment.

## 2. VS Code Extension Internals

The VS Code Extension (`antimatter_extension`) is the beating heart of the system. It is implemented in TypeScript and compiles to a lightweight CommonJS bundle via `esbuild`. 

### 2.1 API Boundaries and Integration Points

The extension does not interact directly with the Google AntiGravity backend servers. Instead, it acts as a "sidecar" to the official AntiGravity IDE extension, inferring agent state by observing the filesystem and executing commands through VS Code's standard inter-process communication (IPC) channels.

#### 2.1.1 Log Tailing and Trajectory Construction
AntiGravity agents log their execution trajectory (thoughts, tool calls, tool responses, and standard output) into a `transcript.jsonl` file located within a hidden `.gemini` application data directory. The Antimatter extension employs an aggressive, highly-optimized polling and tailing mechanism:
- **File Descriptors:** Instead of reading the entire file into memory, it maintains a persistent file descriptor and tracks `lastReadBytes`.
- **Buffer Allocation:** On each polling cycle (every 500ms), it uses `fs.statSync` to check for file size increments. If the file has grown, it allocates a minimal `Buffer` matching the byte delta and reads only the newly appended chunks.
- **JSON Lines Parsing:** The raw buffer is decoded as UTF-8, split by newline delimiters, and parsed into discrete JSON objects representing `TrajectoryStep` interfaces.
- **State Hydration:** These steps are cached in a volatile `currentTranscriptHistory` array. When a new mobile client connects, this entire array is flushed to the client to immediately hydrate its UI state, ensuring no context is lost.

#### 2.1.2 Command Injection
To achieve bi-directional remote control, the extension leverages the `vscode.commands.executeCommand` API. When the Android app sends an `InboundMessage` payload (such as a chat message), the extension decodes it and triggers native AntiGravity commands:
- `antigravity.openAgent`: Focuses the agent panel in the IDE.
- `antigravity.sendPromptToAgentPanel`: Injects the remote text prompt directly into the agent's context queue.
- `workbench.action.chat.cancel`: Interrupts the current agent execution loop.

This approach guarantees that Antimatter never violates the state machine of the official extension, as all remote commands are executed precisely as if the user had typed them locally on their keyboard.

### 2.2 WebSocket Server Lifecycle

The extension manages a dedicated `ws.WebSocketServer` bound explicitly to `0.0.0.0:8765`. 
- **Startup:** The server lifecycle is tied to the `activate` event of the extension. It can be configured to auto-start or require manual triggering via the command palette.
- **Client Management:** It maintains a `Set<WebSocket>` of active connections. All state updates (new steps, file changes) are broadcasted iteratively to this set.
- **Teardown:** On VS Code exit or manual stop, the server gracefully closes all active sockets with a standard `1000 Normal Closure` code to inform clients that the disconnection was intentional.

## 3. Networking & Security Design

Exposing a local development environment to the public internet is inherently dangerous. Antimatter implements a defense-in-depth security posture, completely removing legacy protocols like LocalTunnel and raw IP forwarding.

### 3.1 Cloudflare Edge Routing

Antimatter relies exclusively on Cloudflare for ingress routing. This provides automatic DDoS protection, TLS 1.3 encryption, and global Anycast routing.

#### 3.1.1 Managed Zero Trust Tunnels
For advanced users, the extension can be configured with a `cloudflareHostname`. In this mode, the user is expected to run a global `cloudflared` daemon authenticated with their Cloudflare account. The daemon multiplexes traffic from Cloudflare's edge nodes down to the local `8765` port.

#### 3.1.2 Quick Tunnels (Zero-Config Fallback)
To ensure frictionless onboarding, if no hostname is provided, the extension dynamically spawns a localized instance of `cloudflared`. 
- **Subprocess Execution:** It executes `cloudflared tunnel --url http://127.0.0.1:8765 --protocol http2`.
- **Protocol Forcing:** By forcing `http2`, Antimatter circumvents aggressive ISP and corporate firewall rules that commonly block UDP-based QUIC connections (which `cloudflared` attempts to use by default).
- **Log Parsing:** The extension monitors the `stderr` stream of the child process, applying regex pattern matching to extract the dynamically generated `trycloudflare.com` URL. This URL is then instantly piped into the Pairing QR Code generator.

### 3.2 Authentication & Cryptography

A public Cloudflare URL means anyone on the internet can attempt to connect to the WebSocket server. To prevent unauthorized access, Antimatter implements strict token-based authentication.

#### 3.2.1 The Pairing Token
Upon its first initialization, the VS Code extension uses Node's native `crypto` module to generate 32 bytes of secure random data:
`crypto.randomBytes(32).toString('base64url')`
This yields a 256-bit Bearer Token. This token is persistently saved in the `vscode.ExtensionContext.globalState`, ensuring it survives IDE restarts but remains isolated to the specific host machine.

#### 3.2.2 The Handshake Protocol
The token is embedded directly into the Pairing QR Code alongside the Cloudflare URL. When the Android app connects, it appends the token as a query parameter (e.g., `wss://[domain]?token=[token]`).

The WebSocket upgrade handler intercepts the HTTP request before the socket is fully established. It parses the URL, extracts the token, and performs a strict equality check against the `globalPairingToken`.

#### 3.2.3 Connection-Level Rate Limiting
Even with 256 bits of entropy, exposing an authentication endpoint invites brute-force attempts. Antimatter implements an in-memory IP-based rate limiter during the handshake phase:
- A `Map<string, { count: number, resetAt: number }>` tracks failed token attempts per `remoteAddress`.
- If an IP exceeds 5 consecutive failed attempts, it is hard-banned for 60 seconds.
- During a ban, the WebSocket upgrade is immediately rejected with a `4000 Rate Limited` status code, preventing the server from wasting resources on malicious connections.

## 4. Android App Architecture

The Antimatter companion application (`antimatter_app`) is an Android client engineered using Kotlin, Jetpack Compose, and Kotlin Coroutines. It prioritizes reactivity and strict adherence to the MVVM (Model-View-ViewModel) architectural pattern.

### 4.1 UI and State Hoisting

The application employs a unidirectional data flow. The UI layer (Jetpack Compose) is entirely stateless. 
- All mutable state is housed within the `ChatViewModel`, encapsulated within a highly optimized `MutableStateFlow<ChatUiState>`.
- The `ChatUiState` data class represents the absolute source of truth for the entire application, containing the WebSocket connection status, the list of parsed `TrajectoryStep` objects, the active file contents, and error messages.
- Composables (like `ConnectScreen` and `ChatScreen`) observe this `StateFlow` via `collectAsStateWithLifecycle()`. This ensures that UI recompositions are triggered only when the underlying data actually mutates, preventing unnecessary rendering overhead.

### 4.2 Network Client Implementation

The WebSocket connection is managed by OkHttp3. The `BridgeWebSocket` class abstracts the OkHttp lifecycle, translating raw text frames into strongly-typed Kotlin domain objects.

- **Deserialization:** Incoming text frames are parsed using Gson. The parsing logic dynamically maps the untyped `TrajectoryStep` maps to specific UI rendering strategies based on the `case` field (e.g., routing `TOOL_CALL` objects to the tool visualization composables).
- **Concurrency:** The `BridgeWebSocket` utilizes a `SharedFlow` to emit parsed messages. The `ChatViewModel` launches coroutines in the `viewModelScope` to safely collect these emissions on a background thread, pushing the state updates to the main thread seamlessly.

### 4.3 FOSS vs Standard Build Flavors

Antimatter is fiercely committed to open-source principles. The `build.gradle.kts` configuration defines two distinct product flavors: `standard` and `foss`.

- **Standard Flavor:** Includes proprietary SDKs (like Firebase Crashlytics and Google Play Services) for telemetry, analytics, and enhanced push notifications.
- **FOSS Flavor (Free and Open Source Software):** Strips out all proprietary dependencies. This flavor relies exclusively on AOSP-compatible libraries (e.g., ZXing for QR code scanning instead of ML Kit). This guarantees that users prioritizing extreme privacy and data sovereignty can compile and deploy the app without transmitting any telemetry to third-party tracking networks.

## 5. WebSocket Protocol Specification

The bi-directional communication between the VS Code Extension and the Android App follows a strictly defined, JSON-based payload schema. Every message consists of a `type` field that dictates the parsing strategy for the remainder of the payload.

### 5.1 Outbound Messages (Bridge → Client)

The server broadcasts state updates and trajectory steps to connected clients.

#### `SESSION_STATE`
Sent immediately upon a successful connection handshake. Hydrates the client with the current metadata of the AntiGravity agent.
```json
{
  "type": "SESSION_STATE",
  "conversationId": "uuid-string",
  "model": "gemini-2.5-pro",
  "stepCount": 142,
  "cloudflareUrl": "wss://tunnel.domain.com"
}
```

#### `STEP`
Represents an incremental change in the agent's thought process or execution cycle. Sent continuously during generation.
```json
{
  "type": "STEP",
  "index": 143,
  "step": {
    "case": "TOOL_CALL",
    "tool": "write_to_file",
    "value": "{\n  \"TargetFile\": \"/path/to/file.ts\"\n}"
  }
}
```

#### `ACTIVE_FILE` & `FILE_CONTENT`
Broadcasted when the user switches tabs in their VS Code editor, allowing the Android app to mirror the developer's current focus area.
```json
{
  "type": "ACTIVE_FILE",
  "path": "/home/user/project/src/main.ts",
  "language": "typescript"
}
```

### 5.2 Inbound Messages (Client → Bridge)

The Android app sends commands to the server to mutate the IDE state or inject instructions into the agent.

#### `SEND_MESSAGE`
Injects a text prompt into the AntiGravity panel, effectively allowing the user to converse with the agent remotely.
```json
{
  "type": "SEND_MESSAGE",
  "text": "Refactor this function to use Kotlin Coroutines."
}
```

#### `CANCEL_RESPONSE`
Triggers the `workbench.action.chat.cancel` IPC command, halting the current LLM generation stream or tool execution loop.
```json
{
  "type": "CANCEL_RESPONSE"
}
```

## 6. Future Scalability & Edge Cases

The Antimatter architecture was designed with robustness in mind, but autonomous agents introduce unique scalability challenges that the protocol accommodates.

### 6.1 Backpressure and Chunking
A rogue AI agent can occasionally generate massive payload outputs (e.g., logging a 50MB database dump to standard output). If the VS Code extension attempts to broadcast a 50MB JSON object over a single WebSocket frame, the OkHttp client on Android will trigger an Out-Of-Memory (OOM) exception. 
To mitigate this, the extension employs **Log Chunking**. When tailing the `transcript.jsonl` file, it batches `TrajectoryStep` objects into chunks of 200. This ensures that memory consumption remains flat, allowing the Android UI thread to gracefully process incoming frames without locking the display.

### 6.2 Connection Resiliency
Mobile networks are inherently unstable. The `BridgeWebSocket` implements aggressive exponential backoff strategies for reconnection. If the Cloudflare tunnel drops packets due to network degradation, the Android app will seamlessly attempt to reconnect, requesting the `SESSION_STATE` payload to calculate the differential between its localized `stepCount` and the server's authoritative state, ensuring no thoughts or tool calls are ever missed in the UI.
